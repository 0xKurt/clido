# Alfred V1.5 Plan

## Goal

Make Alfred significantly better to operate every day without expanding the architecture.

V1.5 is about operational quality, automation readiness, and confidence. It should feel meaningfully smoother, safer, and cheaper to run than V1.

## Why These Features Do Not Belong In V1

V1 proved that Alfred can complete real tasks. That is a separate problem from proving that Alfred is safe and efficient to run repeatedly.

These features address the second problem:
- you cannot safely automate Alfred until its output is machine-readable
- you cannot trust Alfred on sensitive repos until it detects secrets
- you cannot control cost without tracking it
- you cannot diagnose setup problems without a health check

V1 deliberately omitted these to keep its scope achievable. V1.5 closes those gaps.

## Included Scope

From the main roadmap:
- Phase 4.4 — Cost Tracking
- Phase 4.6 — Parallel Tool Execution with Bounded Concurrency
- Phase 6.2 — Context Efficiency
- Phase 7.2 — Secret Detection
- Phase 8.2 — JSON and Stream-JSON Output
- Phase 8.4 — `alfred doctor`

## Why These Features Come Next

These additions have immediate operational value:
- cost tracking prevents silent runaway spend and makes Alfred usable in budget-constrained workflows
- bounded concurrency speeds up read-heavy tasks while preventing runaway parallelism
- context efficiency reduces repeated-token waste on long sessions
- secret detection lowers the risk of accidental leakage when writing files
- machine-readable output (`--output-format json` and `--output-format stream-json`) unlocks scripting, CI, and downstream tooling
- `alfred doctor` makes setup and support dramatically easier

## What V1.5 Should Feel Like

Compared to V1, Alfred should now:
- be usable from scripts and CI-style environments without manual output parsing
- be faster on read-heavy tasks due to concurrent tool execution
- be safer on repositories containing secrets or credentials
- show the operator exactly what was spent and what was executed
- be easier to debug when something goes wrong at setup

## Explicitly Out of Scope

Still not required for V1.5:
- multi-provider support
- subagents
- planner / task graph
- memory system
- repository indexing
- MCP support
- full OS-level sandboxing
- production packaging and distribution

## Exit Criteria

V1.5 is done when all of the following are true:
- Alfred reports cumulative token cost at the end of every session
- repeated reads of unchanged files do not add duplicate content to context
- batches of read-only tool calls execute concurrently under a configurable semaphore cap
- secret-like patterns in Write or Edit content produce a clear warning before the write proceeds
- `--output-format json` produces a stable, well-defined JSON result on completion
- `--output-format stream-json` produces valid newline-delimited JSON for every event
- `alfred doctor` detects the most common local setup problems and reports them clearly

## Suggested Internal Milestones

1. Implement cost tracking with pricing table and session summary.
2. Add `--output-format json` and `--output-format stream-json` with stable schemas.
3. Implement concurrent read-only tool execution with semaphore.
4. Add context deduplication for repeated reads of unchanged files.
5. Implement secret-pattern scanning for Write and Edit content.
6. Build `alfred doctor` with provider, config, and storage checks.
7. Validate all output contracts and operator workflows end to end.

## Release Intent

This is the release that makes Alfred comfortable for repeated personal use and safe enough for early automation pipelines.
