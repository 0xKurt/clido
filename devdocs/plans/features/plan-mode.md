# Feature: Interactive Plan Mode

## Problem Statement

Claude Code has a `plan` permission mode where the agent drafts a plan before executing.
The problem: it is a blunt on/off gate — the agent writes a plan as free-form text in the
conversation, the user can only say "go ahead" or abandon. There is no structured editing,
no ability to reorder steps, delete a step, or inject constraints before execution begins.
The plan lives only in the conversation — it cannot be saved, versioned, or reused.

Clido already has a `--planner` flag that decomposes prompts into a JSON task graph (DAG),
but it fires silently and the user has no way to inspect or change the plan before execution.

**This feature builds a first-class interactive plan mode that is meaningfully better than
every competitor's implementation.**

---

## Competitive Analysis

| Product | Plan support | Editable? | Structured? | Saveable? |
|---------|-------------|-----------|-------------|-----------|
| Claude Code | `--plan` flag: text plan, user says go/stop | No | No | No |
| Cursor | Inline "thinking" shown, no pre-approval | No | No | No |
| Cline | Shows step-by-step intent, approve each step | Per-step only | No | No |
| Aider | `/architect` mode: free-text plan | No | No | No |
| **clido** (target) | Full TUI plan editor, structured DAG, editable, saveable | **Yes** | **Yes** | **Yes** |

---

## Our Improvements Over Competitors

1. **Structured plan** — Not free-form text but a proper task graph (id, description, depends_on, estimated_complexity). The model produces JSON; clido validates and renders it.
2. **Full TUI editor** — Before execution begins, the user sees the plan as an editable list. They can: rename a task, delete a task, add a new task, reorder tasks, mark a task as `skip`, add notes/constraints to a task.
3. **Plan persistence** — Plans are saved to `.clido/plans/<plan-id>.json`. They can be listed (`clido plan list`), re-run (`clido plan run <id>`), and shared.
4. **Execution progress overlay** — During execution, the plan is shown as a live checklist in the TUI status area: ✓ done, ↻ running, ○ pending, ✗ failed. Not just a spinner.
5. **Mid-execution pause** — User can press `p` during plan execution to pause after the current step, inspect, edit the remaining plan, and resume.
6. **Plan templates** — Common plans (e.g. "add a new API endpoint", "refactor a module") can be saved as templates and reused.
7. **Dry-run mode** — `clido --plan --dry-run` generates and shows the plan but never executes.

## Design Decisions

### Plan JSON Schema

```json
{
  "id": "plan-abc123",
  "goal": "Refactor the auth module to use JWT",
  "created_at": "2026-03-21T20:00:00Z",
  "tasks": [
    {
      "id": "t1",
      "description": "Read existing auth module and identify all session-based patterns",
      "depends_on": [],
      "complexity": "low",
      "skip": false,
      "notes": "",
      "status": "pending"
    },
    {
      "id": "t2",
      "description": "Add jwt crate dependency to Cargo.toml",
      "depends_on": ["t1"],
      "complexity": "low",
      "skip": false,
      "notes": "",
      "status": "pending"
    },
    {
      "id": "t3",
      "description": "Implement JWT token generation in auth/token.rs",
      "depends_on": ["t2"],
      "complexity": "high",
      "skip": false,
      "notes": "Use HS256 algorithm, expiry = 24h",
      "status": "pending"
    }
  ]
}
```

`status` values: `pending` | `running` | `done` | `failed` | `skipped`.
`complexity` values: `low` | `medium` | `high` — used for estimated time display only.

### Plan generation

The plan is generated by the LLM via `AgentLoop::complete_simple()` with a structured
prompt asking for JSON output. The existing `clido_planner::parse_plan()` parses and
validates the graph. On parse failure, fall back to a single-task plan (same as today).

### Plan editing (TUI)

When `--plan` is active and a plan is generated, execution is paused. The TUI enters
**plan edit mode** — a full-screen overlay (not a small modal) showing:

```
╔══ Plan: Refactor auth module to JWT ══════════════════════════════════╗
║  3 tasks · estimated complexity: medium                                ║
║                                                                        ║
║  ○ t1  Read existing auth module                          [low]  [   ] ║
║  ○ t2  Add jwt crate to Cargo.toml            (needs t1)  [low]  [   ] ║
║  ● t3  Implement JWT token generation         (needs t2)  [high] [   ] ║  ← selected
║                                                                        ║
║  [Enter] Edit task  [d] Delete  [n] New task  [Space] Skip             ║
║  [↑↓] Move  [r] Reorder deps  [s] Save plan  [x] Execute  [Esc] Abort ║
╚════════════════════════════════════════════════════════════════════════╝
```

Selecting a task and pressing Enter opens an inline edit form:
```
  Description: [Implement JWT token generation         ]
  Notes:        [Use HS256 algorithm, expiry = 24h     ]
  Complexity:   ○ low  ● medium  ○ high
  Skip:         [ ]
  [Enter] Save   [Esc] Cancel
```

### Plan execution overlay

During execution, the plan checklist is shown in the TUI sidebar / status area:

```
  Plan progress (2/3)
  ✓ t1  Read auth module                14s
  ↻ t2  Add jwt crate               (running…)
  ○ t3  Implement JWT token
```

`p` pauses after the current step. The paused state shows remaining tasks as editable
(same editor as above). `x` resumes.

### Plan persistence

Plans are stored as JSON in `.clido/plans/`. On `clido plan run <id>`, the saved plan
is loaded and execution starts from the first `pending` task (supports resuming failed plans).

## Crate and File Implementation

### `crates/clido-planner/src/`

**Extend `graph.rs`:**
- Add `complexity: Complexity` field to `TaskNode`
- Add `skip: bool` field to `TaskNode`
- Add `notes: String` field to `TaskNode`
- Add `status: TaskStatus` enum field
- Add `PlanMeta { id, goal, created_at }` struct
- Add `Plan { meta: PlanMeta, tasks: Vec<TaskNode> }` — top-level saveable plan

**Extend `parser.rs`:**
- Parse new fields; backward compatible (unknown fields ignored)
- `parse_plan_with_meta(text: &str) -> Result<Plan>` — new entry point
- `plan_to_json(plan: &Plan) -> String` — serialize back to JSON for saving/editing

**New `editor.rs`:**
- `PlanEditor` struct: in-memory editing operations (rename, delete, add, reorder, set_skip, set_notes)
- All operations return `Result<(), PlanEditError>` and re-validate the DAG after each change
- No I/O — purely functional, testable without TUI

**New `storage.rs`:**
- `save_plan(workspace_root: &Path, plan: &Plan) -> Result<PathBuf>`
- `load_plan(workspace_root: &Path, id: &str) -> Result<Plan>`
- `list_plans(workspace_root: &Path) -> Result<Vec<PlanSummary>>`
- `delete_plan(workspace_root: &Path, id: &str) -> Result<()>`
- Plans stored at `<workspace>/.clido/plans/<id>.json`

### `crates/clido-cli/src/tui.rs`

**New `App` state:**
- `plan_edit_mode: bool` — when true, full-screen plan editor is active
- `edit_plan: Option<Plan>` — plan being edited
- `plan_selected_task: usize` — currently highlighted task in editor
- `plan_task_editing: Option<TaskEditState>` — inline form state (description, notes, complexity)
- `plan_execution_view: Option<Vec<TaskStatus>>` — live status during execution

**New `AgentEvent` variant:**
- `PlanReady { plan: Plan }` — plan generated, awaiting user review
- `PlanTaskStarted { task_id: String }` — executor started a task
- `PlanTaskDone { task_id: String, success: bool }` — task complete
- `PlanPaused` — user paused mid-execution

**New rendering functions:**
- `render_plan_editor(f: &mut Frame, app: &App, area: Rect)` — full-screen plan edit overlay
- `render_task_edit_form(f: &mut Frame, state: &TaskEditState, area: Rect)` — inline form
- `render_plan_execution_sidebar(f: &mut Frame, plan: &Plan, area: Rect)` — live checklist

**New key handling:**
- When `plan_edit_mode == true`:
  - `↑`/`↓` — move selection
  - `Enter` — open task edit form
  - `d` — delete selected task (with confirmation)
  - `n` — new task (appended, then edit form opens)
  - `Space` — toggle skip
  - `r` — reorder mode (move task up/down in execution order)
  - `s` — save plan to `.clido/plans/`
  - `x` — execute plan (exit plan edit mode, start execution)
  - `Esc` — abort (do not execute)
- During execution:
  - `p` — pause after current task

**New slash commands:**
- `/plan` (already exists — extend): shows plan checklist if plan is active, or shows last plan
- `/plan edit` — re-enter plan editor for the current plan
- `/plan save` — save current plan to file
- `/plan list` — list saved plans

### `crates/clido-cli/src/plan_cmd.rs` (new file)

- `run_plan_list(workspace_root)` — list saved plans with id, goal, task count, status
- `run_plan_show(workspace_root, id)` — show plan details
- `run_plan_run(workspace_root, id, cli)` — load plan and run it (skip completed tasks)
- `run_plan_delete(workspace_root, id)` — delete plan

### `crates/clido-cli/src/cli.rs`

New subcommand:
```
Plan {
    cmd: PlanCmd,
}
```

```rust
pub enum PlanCmd {
    /// List saved plans.
    List,
    /// Show a saved plan.
    Show { id: String },
    /// Run a saved plan.
    Run { id: String },
    /// Delete a saved plan.
    Delete { id: String },
}
```

Extend existing flags:
- `--plan` (already exists as `--planner`) — rename/alias to `--plan` for discoverability; keep `--planner` as alias
- `--dry-run` — generate plan, show in TUI editor, do not execute even if user presses `x`. Exit after showing.

### `crates/clido-agent/src/agent_loop.rs`

- `run_with_plan(plan: &Plan, session, pricing, cancel) -> Result<String>` — executes tasks in topological order using existing `PlanExecutor`; emits `PlanTaskStarted`/`PlanTaskDone` events
- Pause support: `pause_requested: Arc<AtomicBool>` checked between tasks

## CLI Surface

### New subcommand: `clido plan`

```
clido plan list                     # list all saved plans
clido plan show <id>                # show plan tasks and status
clido plan run <id>                 # execute saved plan (resumes if partially done)
clido plan delete <id>              # remove plan
```

### Modified flags:

```
clido --plan "refactor auth"        # generate plan, open TUI editor, execute on approval
clido --plan --dry-run "..."        # generate plan, open editor, never execute
clido --plan --no-edit "..."        # generate plan, skip editor, execute immediately (CI-friendly)
```

### Help text:

```
--plan              Enable plan mode: decompose the task into an editable step-by-step
                    plan before execution. Opens the plan editor in the TUI. Use
                    --no-edit to skip the editor (for scripts/CI).
--dry-run           With --plan: generate the plan and open the editor, but never execute.
                    Useful for reviewing what the agent would do.
--no-edit           With --plan: skip the interactive editor and execute immediately.
```

## TUI Changes

### Plan edit mode (full-screen overlay)

- Triggered when `AgentEvent::PlanReady` is received
- Blocks all other input (no chat input, no slash commands except Esc to abort)
- Full-screen overlay using `Clear` widget + `Block` with double-line border
- Tasks rendered as a scrollable list (handle plans > terminal height)
- Selected task highlighted with cyan background
- Task in edit form: text input fields using crossterm raw mode
- Complexity shown as colored badge: `[low]` dim, `[med]` yellow, `[high]` red
- `skip` shown as strikethrough on task description

### Plan execution sidebar

- When executing a plan, the right 30% of the TUI shows the plan checklist
- Main chat area shrinks to 70%
- Live tick on the running task (spinner prefix)
- Completed tasks show elapsed time
- Failed tasks shown in red with `✗`

### Status strip

- During plan execution: `Plan: 2/5 tasks  ●  t3 running (14s)`

## Config / Schema

### New `[plan]` section in `config.toml`:

```toml
[plan]
# Always use plan mode (equivalent to always passing --plan).
always_plan = false

# Skip the interactive editor and execute immediately.
# Useful for non-TTY / CI environments.
no_edit = false

# Maximum number of tasks in a generated plan.
max_tasks = 10

# Minimum complexity to trigger plan mode automatically.
# "high" = only for complex prompts, "any" = always.
auto_plan_threshold = "off"

# Save every generated plan to .clido/plans/ automatically.
auto_save = false
```

### `AgentConfig` additions:

```rust
pub use_plan_mode: bool,          // --plan flag
pub plan_no_edit: bool,           // --no-edit flag
pub plan_dry_run: bool,           // --dry-run flag
```
